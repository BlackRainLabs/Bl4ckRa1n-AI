Only use `mode: push` when the user explicitly requests it.\n- Always verify the new commit SHA after every commit or push.\n- Never proceed if `git status` shows no changes.\n- Never force-push to protected branches unless `force: true` is set **and** the user has confirmed.\n- Log every action: timestamp, repo, files, mode, and result.\n\n## Handling User Requests\n\nUsers often speak casually. Translate these natural requests into proper actions:\n\n- \"Create a PowerShell script about X and upload it\" → Create file + `mode: pr`\n- \"Check this script and improve it\" → Review → propose changes + `mode: pr`\n- \"Just push this\" / \"Commit it directly\" → `mode: push`\n- \"Fix the bug and deploy it\" → Fix + `mode: pr`\n- \"Build a module and put it in HermesDev\" → Create + `mode: pr`\n\nAlways use clear, professional commit messages even when the user is casual. Confirm before risky actions.\n\n## Parameters\n\n- `target_repo` (required)\n- `files` (required) — array of paths\n- `commit_message` (required)\n- `mode` — `pr` (default) or `push`\n- `base_branch` — default `main`\n- `pr_title` (recommended)\n- `dry_run` — `true` for testing\n- `force` — only when user explicitly allows\n\n## Workflows\n\n**Standard (Recommended):**\n1. Create feature branch (`agent/YYYYMMDD-HHMMSS`)\n2. Make changes\n3. Commit with clear message\n4. Push branch\n5. Create Pull Request via `gh pr create`\n6. Verify new SHA and report\n\n**Direct Push (Restricted):**\nOnly when user explicitly requests `mode: push`. Follow same steps but push directly to target branch.\n\n**Large Files:**\nUse `cp` + `sed` cleanup when needed, then follow Standard workflow.\n\n## Pitfalls to Avoid\n\n- Do **not** default to direct push.\n- Do **not** skip SHA verification.\n- Do **not** push to protected branches without confirmation + `force: true`.\n- Do **not** proceed with empty commits.\n- Do **not** ignore branch protection rules.\n- Do **not** rush — prefer Pull Requests.\n\nIf any rule would be violated, stop and ask for user confirmation.\n\n## Decision Process\n\n1. Understand the request and required changes\n2. Choose safest `mode` (default = `pr`)\n3. Select clear `commit_message`\n4. Execute workflow\n5. Verify result (new SHA + success)\n6. Report outcome clearly\n\n## Best Practices\n\n- Prefer Pull Requests in almost all cases\n- Use `dry_run: true` when unsure\n- Write descriptive commit messages\n- Log every deployment\n- Use GitHub App authentication when available\n\n**Workdir Note:** Default to `/home/null/git-work/Bl4ckRa1n-AI` (Bl4ckRa1n-AI clone).
