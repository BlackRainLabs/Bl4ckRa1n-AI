Use `mode: pr` only when explicitly requested.\n- Always verify the new commit SHA after every commit or push.\n- Never proceed if `git status` shows no changes.\n- Never force-push to protected branches unless `force: true` is set **and** the user has confirmed.\n- Log every action: timestamp, repo, files, mode, and result.\n\n## Handling User Requests\n\nUsers often speak casually. Translate these natural requests into proper actions:\n\n- \"Create a PowerShell script about X and upload it\" → Create file + direct push to main\n- \"Check this script and improve it\" → Review → changes + direct push\n- \"Just push this\" / \"Commit it directly\" → direct push to main\n- \"Fix the bug and deploy it\" → Fix + direct push\n- \"Build a module and put it in HermesDev\" → Create + direct push\n- \"Make a PR\" → `mode: pr`\n\nAlways use clear, professional commit messages even when the user is casual. Confirm before risky actions.\n\n## Parameters\n\n- `target_repo` (required)\n- `files` (required) — array of paths\n- `commit_message` (required)\n- `mode` — `pr` (only for PRs) or omit/default: direct push\n- `base_branch` — default `main`\n- `pr_title` (recommended for PRs)\n- `dry_run` — `true` for testing\n- `force` — only when user explicitly allows\n\n## Workflows\n\n**Direct Push (Default):** \n1. Checkout/switch to main\n2. Make changes\n3. Commit with clear message\n4. Push to main\n5. Verify new SHA and report\n\n**PR Mode (when mode: pr):** \n1. Create feature branch (`agent/<AGENT_NAME>`)\n2. Make changes\n3. Commit + push branch\n4. Create PR via `gh pr create`\n5. Verify + report\n\n**Large Files:**\nUse `cp` + `sed` cleanup when needed, then follow chosen workflow.\n\n## Pitfalls to Avoid\n\n- Do **not** create branches/PRs unless `mode: pr`.\n- Do **not** skip SHA verification.\n- Do **not** push to protected branches without confirmation + `force: true`.\n- Do **not** proceed with empty commits.\n- Do **not** ignore branch protection rules.\n- **\\n Escapes:** read_file outputs `\\n`. Always `sed -i 's/\\\\n/\\n/g'` before git add.\n\nIf any rule would be violated, stop and ask for user confirmation.\n\n## Decision Process\n\n1. Understand the request and required changes\n2. Default to direct push (fast for solo edits)\n3. Use `mode: pr` only if requested\n4. Select clear `commit_message`\n5. Execute workflow\n6. Verify result (new SHA + success)\n7. Report outcome clearly\n\n## Best Practices\n\n- Direct push for most cases\n- PRs only for reviews/collabs\n- Use `dry_run: true` when unsure\n- Descriptive commits\n- Log deploys\n- GitHub App auth when possible\n\n**Workdir:** `/home/null/git-work/Bl4ckRa1n-AI`.
